解析：MyBatis 将 XML 映射文件解析并构建为 MappedStatement 对象。

生成statement的全过程：

第一步：扫描与读取

MyBatis 根据配置 mapper-locations: classpath*:mapper/*.xml，找到所有的 XML 文件，并将它们读入内存，解析成 XML 文档树。

第二步：解析标签 (Parse Tags)

MyBatis 遍历 XML 中的每一个标签（如 <select>, <insert>, <update>, <delete>）。

```java
<select id="selectById" resultType="com.example.User">
    SELECT * FROM user WHERE id = #{id}
</select>
```

第三步：构建MapperStatement

MyBatis 会为上面的 <select> 标签创建一个 MappedStatement 对象。这个对象里存了什么？

ID: com.example.mapper.UserMapper.selectById (命名空间 + 标签ID)

SQL Source: 解析后的 SQL 对象（处理了 ${} 和 #{} 占位符逻辑）。

Command Type: SELECT。

Parameter Map: 知道 #{id} 对应方法参数的 id。

Result Map: 知道查询结果要转换成 com.example.User 类。

第四步：注册到 Configuration (Register)

```java
// 伪代码
configuration.addMappedStatement("com.example.mapper.UserMapper.selectById", mappedStatementObj);
```

第五步：绑定 Mapper 接口 (Bind Interface)

MyBatis 使用 JDK 动态代理，为 UserMapper 接口生成一个代理对象。当你调用 userMapper.selectById(1L) 时：代理对象拦截这个方法调用。它拿着方法名 selectById 去那个“大地图”里找对应的 MappedStatement。找到后，拿出里面的 SQL 和配置，去数据库执行。

逻辑如下

![](images/WEBRESOURCEc1b4a0e79224427f93f9f00f664d1e5bimage.png)

如果不生成statement，每次调用都要重新解析XML文本，速度非常慢

生成statement的意义在于

1. 性能优化：XML只在启动的时候解析一次，生成内存对象。运行时直接拿对象执行，速度飞快

1. 类型安全：在启动阶段，如果XML写错了（比如字段名不对、类名找不到），MyBatis会在生成Statement时直接报错，让你启动失败，而不是等到运行时报错

1. 解耦：让Java接口和XML SQL通过ID动态关联起来，不需要硬编码SQL字符串在Java代码里

总结：生成statement，就是Mybatis在启动时，把写的XML里的死SQL模版，加工成了内存里的活的执行指令对象（MapperStatement），方便你的Java代码能随时调用它去查数据库

这里为什么一定要用updateWrap，而不是"先查后减"

**问题一：并发竞争**

场景设定：以扣200元为例。用户id = 1；初始余额为1000元；有两个请求（比如两个用户同时下单），都要从这个账户扣200元（这里的两次请求都应该合法扣款——1000 -200 -200 = 600元）

如果使用“先查后减”，可能得到错误的结果,比如800元

并发竞争原理：

```java
// 假设这是在一个方法里
User user = userMapper.selectById(1L);        // ① 查
user.setBalance(user.getBalance() - 200);     // ② 改
userMapper.updateById(user);                  // ③ 写
```

两个并发的线程的执行流程

| 时间 | 线程 A（请求1） | 线程 B（请求2） | 
| -- | -- | -- |
| t1 | 执行 | （还没开始） | 
| t2 | （计算中） | 执行 | 
| t3 | 执行 | （计算中） | 
| t4 | （完成） | 执行 | 


最终的结果是id-1的user的balance = 800；

正确的做法是数据库的原子更新：

```java
UPDATE user SET balance = balance - 200 WHERE id = 1;
```

**问题二：事务可以解决吗？**

事务不能解决这个问题。因为事务的默认隔离级别是**READ COMMITTED**（读已提交），它的行为是“每次select都会读取最新的已提交数据”。但是在“先查后减”中，线程a、b都在t1时刻读到了1000，都提交的balance是800；

如何用事务+锁来解决？

将事务的隔离级别设置为悲观锁。这样性能就很差了

```java
@Transactional
public void deduct(Long userId) {
    User user = userMapper.selectByIdForUpdate(userId); // 加锁
    user.setBalance(user.getBalance() - 200);
    userMapper.updateById(user);
}
```
